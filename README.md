# JARVIS — a local, agentic desktop assistant

A Jarvis-style AI that controls your Windows computer: it takes a screenshot,
reads what's on screen, **thinks**, and **acts** — moving the mouse, clicking by
coordinate, typing, launching apps, running commands, and working with files —
looping until your task is done. It runs a **small local model** (via Ollama) so
it fits a modest machine (built and tuned for an RTX 2050 / 4 GB VRAM / 8 GB RAM).

```
  perceive  ──▶  think  ──▶  act  ──▶  observe result  ──▶  (repeat)
 screenshot     local LLM    mouse /      feed back to
 + UI tree      picks one    keyboard /   the model
                action       apps / OS
```

This repo contains the three things you asked for, wired together end-to-end:

1. **The Jarvis app** — `jarvis/`, launched by `run.py`.
2. **A Jarvis-specific dataset** — generated by `dataset/build_dataset.py`.
3. **A training pipeline** — `training/` fine-tunes a base model on that dataset
   and serves the result back to the app through Ollama.

---

## Why this design (read this — it's the important part)

**Agentic screen control needs *grounding*: knowing the exact pixel to click.**
Asking a tiny model to guess a pixel from raw screenshot bytes is unreliable
below ~7B vision models — and a 7B vision model won't fit in 4 GB of VRAM.

So Jarvis does the grounding with **classical tools, not the model**:

- Windows **UI Automation** (and optional OCR) detect the real on-screen
  controls and their exact bounding boxes.
- Each control is labelled with a number ("Set-of-Marks"):

  ```
  [0] Button   "Save"                 @ (512,40)
  [1] Edit     "File name:"           @ (300,80)
  [2] MenuItem "File"                 @ (20,30)
  ```

- The model only has to answer *"click element 0"*. A 1.5–3B **text** model
  does that reliably.

This is what makes accurate desktop control possible on your hardware, and it's
why the fine-tune targets a small text model rather than a giant vision one.
(A raw-vision mode is still included — set `brain.use_vision: true` — for when
you plug in a vision model or a cloud brain.)

## Quick start

### 1. Install the app dependencies
```bash
pip install -r requirements.txt
```
Most were already present on the target machine; `python run.py --check` tells
you what's missing.

### 2. Install a local model brain (Ollama)
Jarvis defaults to a local model served by [Ollama](https://ollama.com/download).
Install it (it auto-starts a server), then pull a small model:
```bash
ollama pull ornith:9b
# alternative options:
# ollama pull qwen2.5:3b-instruct
```

### 3. Run it
```bash
python run.py                       # interactive console
python run.py "open notepad and type Hello, then save as hello.txt"
python run.py --check               # verify environment + backend
```

### Run it from anywhere with the `jarvis` command

To start Jarvis by typing `jarvis` in any terminal (from any directory) instead
of `python run.py`, add this project's `bin/` folder to your PATH once:

```powershell
# PowerShell (persists for your user; open a NEW terminal afterwards)
$bin = "$PWD\bin"
$p = [Environment]::GetEnvironmentVariable('Path','User')
if (($p -split ';') -notcontains $bin) {
    [Environment]::SetEnvironmentVariable('Path', "$p;$bin", 'User')
}
```

Then, in a **new** terminal:
```bash
jarvis                       # interactive console, from any directory
jarvis "open notepad and type Hello"
jarvis --check               # verify environment + backend
```

`bin/jarvis.bat` is self-locating (`%~dp0`), so the command keeps working even
if you move the project — just re-point PATH at the new `bin/` folder. It prefers
the Windows `py` launcher and falls back to `python`.

Inside the console:
```
you > open the calculator and compute 24 times 7
you > :confirm on          # ask before every action (recommended at first)
you > :voice on            # Gemini speech input + Gemini TTS output
you > :help
```

### Gemini voice mode

Voice output uses the dedicated `gemini-3.1-flash-tts-preview` model through
Vertex AI. It is configured separately from the thinking model, so
`brain.model: gemini-3.5-flash` remains unchanged.
Voice transcription uses the lower-latency `gemini-2.5-flash-lite` model with
thinking disabled and a shorter end-of-speech delay.

Authenticate with Application Default Credentials before using voice mode:

```bash
gcloud auth application-default login
```

The TTS model, prebuilt voice, and language can be changed under `voice:` in
`config.yaml`, or with `JARVIS_TTS_MODEL`, `JARVIS_TTS_VOICE`, and
`JARVIS_TTS_LANGUAGE`. The transcription model can be overridden with
`JARVIS_STT_MODEL`. Launch with `python run.py --voice` or use `:voice on`.

> **Safety:** start with `:confirm on` (or `python run.py --confirm`) so you
> approve each action while you learn how the model behaves. Shell commands
> matching a denylist (format, del /, shutdown, …) are refused; file writes are
> sandboxed to your home directory.

## How the app is organised

| Path | What it does |
|------|--------------|
| `jarvis/tools/schema.py` | **The action space** — single source of truth. The app, dataset, and prompts all read it. |
| `jarvis/perception/` | Screenshot (`screen.py`), UI-tree/OCR element detection (`elements.py`), numbered-mark overlay (`annotate.py`). |
| `jarvis/agent/brain.py` | Pluggable LLM backends: **ollama** (default), llamacpp/openai-compatible, anthropic. |
| `jarvis/agent/prompts.py` | System prompt + tolerant parser for the model's JSON action. |
| `jarvis/agent/loop.py` | The perceive→think→act loop. |
| `jarvis/agent/trajectory.py` | Logs every real run as training data (`dataset/data/trajectories/`). |
| `jarvis/tools/registry.py` | Resolves element→coordinate and executes actions. |
| `jarvis/utils/voice.py` | Gemini TTS playback plus microphone recording for Gemini transcription. |
| `jarvis/console.py`, `run.py` | Console REPL and CLI launcher. |

Configuration lives in `config.yaml` (env vars `JARVIS_*` override it).

## The dataset

`dataset/build_dataset.py` generates a fine-tuning dataset **in the app's exact
runtime format** (it imports the same prompt/observation/decision formatters the
live agent uses, so there is zero train/inference skew).

```bash
python dataset/build_dataset.py --n 1500
```
Produces `dataset/data/jarvis_sft.{train,val}.jsonl` (chat/`messages` format),
a 15-line `sample`, and `stats.json`. The current build: **5,860 examples** across
open-app / notepad-write-and-save / web-search / calculator / file-explorer /
scroll-to-find / run-command / clipboard / ask-when-ambiguous scenarios, with
randomised names, text, coordinates and distractor elements.

As you *use* Jarvis, real runs are appended to `dataset/data/trajectories/` —
mix those into the next training run to make it better at your actual tasks.
See `dataset/README.md`.

## Training your own Jarvis brain

Fine-tune a small base model into a `jarvis` model, then point the app at it.
See `training/README.md` for the full walkthrough (including the free-Colab route
if 4 GB VRAM is too tight for a 3B QLoRA).

```bash
pip install -r training/requirements-train.txt        # separate env; big deps
python training/train_qlora.py --model unsloth/Qwen2.5-3B-Instruct --merge
python training/eval_adapter.py --n 100               # sanity-check quality
# convert to GGUF (see training/README.md), then:
python training/export_ollama.py --gguf <path>.gguf --create
# finally:
python run.py --model jarvis
```

  backend and produces valid, correctly-formatted actions.
- ⚠️ **That small run is a proof-of-pipeline, not a production brain.** On the
  60-example eval it emits valid action JSON but grounds unreliably (right on
  simple tasks like "open notepad", wrong on harder element-selection steps).
  This is expected for 0.5B / 600 examples / 1 epoch. To get a genuinely capable
  Jarvis brain, run the **full** dataset for several epochs on a bigger base
  (1.5–3B) — needs more RAM than the 8 GB here (it thrashes swap) or a free
  Colab T4. Everything to do that is built and documented in `training/`.
  - 💡 For a capable assistant **right now**, run the app on a stock model
  (`ollama pull ornith:9b`, or any `hf`/`openai` backend) — the
  accessibility-grounded design means even an untuned model clicks accurately.

### Run your trained model
```bash
# no Ollama needed - loads the LoRA adapter directly:
python run.py --backend hf --model Qwen/Qwen2.5-0.5B-Instruct \
              --adapter training/outputs/jarvis-lora --confirm "open notepad"
# (or set backend: hf + adapter_path in config.yaml)
```
